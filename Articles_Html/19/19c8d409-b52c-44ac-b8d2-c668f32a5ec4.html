What to Do

Document the network and prepare the documentation for PCI DSS compliance. PCI DSS refers to the network documentation as the "firewall and router configuration standards."

Why

The network documentation is very important for designing and implementing information security procedures.&nbsp;Because of&nbsp;this, PCI requires the network documentation to include at least a functional minimum to achieve DSS compliance.

When

Ideally the network documentation should be prepared when the network infrastructure is being designed and&nbsp;should be&nbsp;updated whenever changes to the network are made.&nbsp;What really matters is that the network documentation is available, current,&nbsp;and includes relevant information.&nbsp;The network documentation should be available to maintain PCI DSS compliance.

How

The network documentation is very important for network maintenance, security design and incident response tasks. Proper network documentation is required to achieve PCI DSS compliance. This guide will explain the minimum level of required documentation for PCI DSS compliance and is a good starting point for preparing documentation for your network. Once you have the required minimum, feel free to expand by including whatever information you think is important for network management and security tasks. While preparing the network documentation, it may be useful to have a copy of the company information security policy at hand to make suggestions for policy changes or to reflect the policy in the network documentation. Perform the following actions to prepare the network documentation for PCI DSS compliance:

  1. Define&nbsp;a formal testing process. The network documentation and the information security policy should include a formal testing process for any changes to the firewall and router settings. One way to test changes to the firewall configuration is by performing a detailed port scan of the host(s) protected by the firewall. One way to test changes to the router configuration is by using the ping and traceroute commands common to most network-capable operating systems.

  2. Attach&nbsp;a current network diagram. The network documentation should include an up-to-date network diagram, which shows all the network connections to the cardholder data, including any wireless connections. Acquire a proper diagramming tool. Microsoft Visio is perfectly usable for preparing a network diagram. Draw all network devices within the PCI DSS scope, especially those that store or transmit cardholder data. When all relevant network devices are mapped, draw the network connections between them, including any wireless connections. When the network devices and the connections between them are mapped, draw connections between them and external networks such as the Internet. To maintain the network diagram up-to-date, delegate the responsibility for maintaining the network documentation to a qualified staff member.

  3. Add a requirement for&nbsp;a firewall at each Internet connection and DMZ. The network documentation should require a firewall at each Internet connection and between any DMZ and the internal network. This requirement can be satisfied by including a paragraph in the network documentation that states, "A firewall should be installed at each Internet connection and between any DMZ and the internal network."&nbsp;Start by installing firewalls at each Internet connection.&nbsp;Install firewalls between each DMZ and the internal network.

  4. Add descriptions of groups, roles, and&nbsp;responsibilities. The network documentation should include the descriptions and responsibilities of network management roles. Designate network management roles, such as network administrators, systems administrators, and information security officers. Assign responsibilities to each network management role. Document the descriptions of network roles and their corresponding responsibilities.

  5. Describe all protocols necessary for business. Identify all the protocols necessary for business. To identify the necessary protocols, it may be helpful to identify the network applications used by the business and include the corresponding protocols and ports used by the applications. A web search engine quickly yields typical ports and protocols used by most network applications. If custom port settings are used, which is a good practice to prevent automated attacks, the custom port settings should be documented.

  6. Identify insecure protocols and explain why they are necessary. The network documentation should include a list of all insecure network protocols and explanations of why each is necessary for business operations. Some common Internet protocols have been proven to be insecure, such as FTP, which sends authentication credentials in plain text. Some email protocols may also send authentication information in clear-text, which is often overlooked because email is such an integral part of many business environments. Security extensions are available for email protocols. Verify that each insecure service is necessary and cannot be easily replaced by a secure version, for example by replacing telnet with SSH. If possible, add security measures to each insecure service by using technologies such as VPN, white lists, or application-specific extensions. Make sure to document each insecure service thoroughly, including the nature of the vulnerability and the implemented mitigation measures.

  7. Explicitly authorize the necessary outgoing connections from the cardholder data environment to the Internet.&nbsp;Make a list of necessary outgoing connections from the carholder data environment. One example of important outgoing connections is downloading software updates. Specify local and remote hosts for each necessary outgoing connection. Specify local and remote ports for each necessary outgoing connection.

  8. Document the firewall rule-set. The network documentation should include descriptions and explanations of the restrictions imposed by the firewalls. The firewalls should block all traffic that is not necessary for business operations. For each firewall, document the “deny” rules, including the "deny all."

  9. Add a requirement to review firewall and router configurations at least every six months. The network documentation should include a requirement to review the firewall and router configurations at least every six months. To satisfy this requirement, the documentation may include a paragraph stating, "The firewall and router configurations must be reviewed at least every six months." To make sure the review is performed, delegate responsibility for reviewing the router and firewall configurations to a qualified staff member.
