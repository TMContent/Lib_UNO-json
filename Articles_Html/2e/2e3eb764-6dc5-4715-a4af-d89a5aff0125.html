What to Check For

Ensure log files are protected.

Why

Log files often contain internal application details that could enable an attacker to more effectively target your application.

How to Check

Use these steps to ensure log files are protected:

  1. Verify that the&nbsp;log location is not disclosed. Scan your application's code base to verify that the location of the log files is known only to the internal logging framework.

  2. Check the file permissions of logs. Verify that the logs are set with the appropriate filesystem permissions. All entities, excluding your application and its administrators/developers, must be prohibited access to the logs. If your application backs up its logs, verify that log backups are stored into an environment that is as secure as the storage environment for the current logs.

  3. Check for common logging mistakes. Scan your application's code base for common logging mistakes such as:

    • Logging data structures that contain user-supplied data
    • Appending user-supplied data to an exception's message and/or details
    • Opening logs in "write" mode and not in "append" mode

Note: PCI DSS certification requires that all access to the logs is logged, that logs muct be promptly backed up to a centralized log server or media that is difficult to alter, and logs should be reviewed regularly to watch for suspicious activity that could indicate access by an attacker.

How to Fix

Use the following steps to protect the application's log files:

  1. Avoid disclosing location of logs. The location of the log files should be known only by your application and its administrators. Avoid disclosing the location of the logs to users.

  2. Set appropriate file permissions. Set the filesystem permissions so that only your application and its administrators/developers have access to its logs.

  3. Keep logging framework away from users. Do not allow your application's users to interact directly with the logging framework. The logging framework should log information regarding the performance of the application but should avoid storing user-supplied data inside the logs. Some user-supplied information can aid an attacker into poisoning your application's logs.

  4. Always use the append flag when writing to log files. Make sure that your application always appends new events to the log files. Appending to the logs reduces the risk of overwriting the existing logged events. Example:

    FileOutputStream logFile =
     new FileOutputStream(Environment.getLogPath() + logFilename, true);

  5. Consider using log rotation. Log rotation prevents your application from working with large log files. It allows you to manage and audit your application's logged events without interfering in your application's workings. Use the following steps when utilizing log rotation:

    • Define a rotation period: Schedule your log rotation on a regular basis. For example, rotate your logs every Monday.
    • Create a new log file: Create a new log file when your application needs to rotate its log files. It is recommended to associate the new log file to the period of its creation. For example, the log file corresponding to the week of October 16, 2007 would be named 20071016.log.
    • Back up the existing log files: Back up the existing log files when your application needs to rotate its log files. Consult your internal auditing policy to determine the period for which logs should be backed up. If your application backs up its logs, store them into an environment that is as secure as the storage environment for the current logs.

  6. Consider using an external logging application. Using an external logging application ensures that your application's logs will be kept up-to-date even if your application abruptly terminates.

  7. Consider encoding output that is sent to log files. This can be useful in preventing log forging. Possible encoding mechanisms could include:
    • Stripping out certain characters [\r\n] that are known to create new lines, which can allow an attacker to forge log entries.
    • Encode all log output using HTML encoding, then view it in a browser to see the output.

Note: PCI DSS requirements state that logs should be kept for at least&nbsp;one year.
