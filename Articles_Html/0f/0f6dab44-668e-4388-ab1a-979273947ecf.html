Applies to

  • PCI DSS Requirement 8.5.16

What to Check For

Check that all access to any database containing cardholder data&nbsp;is authenticated. Verify that&nbsp;database access IDs used by applications&nbsp;are not usable by individual users or other applications.

Why

Preventing the database access accounts used by applications from being used interactively or by other processes ensures that if these accounts are compromised, an attacker cannot use them.

How to Check

Use the following steps to check if database access is authenticated:

  1. Confirm user authentication. Review the settings of each database that stores cardholder data and any corresponding applications to make sure that users are authenticated before access.&nbsp;

  2. Verify&nbsp;that only database administrators may issue database queries or have direct access. Get information about connecting to the database from the database administrator. Attempt to log into the database administration interface with valid non-administrator user credentials. Access should be denied to all users except the administrators.

  3. Verify that application access credentials cannot be abused. Make sure that any authentication credentials used by applications to access any database that stores cardholder data can only be used by the corresponding application(s) and not by users or other processes. Get information about connecting to the database from the database administrator. Attempt to log into the database administration interface with valid credentials used by applications. Access should be denied to interactive logons.

How to Fix

Perform the following actions to assure that database access is authenticated:

  1. Implement functionality to authenticate data access. Program the database application to present the user with a login screen, where the user has to be authenticated before being granted access to the information in the database. Use the authentication credentials from the login throughout the database application for access control.

  2. Secure the database administration interface. Allow only the database administrators to access to the database administration interface(s). Usually a database server allows either a terminal session or a visual administration application to connect to a port and issue SQL queries directly. Authentication credentials usually secure this functionality. Do not make access credentials available for any other users besides the database administrator(s).

  3. Secure the database files. To prevent direct access to the database files, use the access control system implemented by the host Operating System to allow access to database administrator(s) only. If possible, block any computer, besides the administrator's workstation, from accessing the database server administration interface by using a white list. A white list is a list of hosts that are exclusively allowed to access some functionality. On UNIX-style systems, white lists are usually configured by editing the /etc/hosts.allow and /etc/hosts.deny files.

  4. Log access to cardholder data. Instrument the application to log access to the cardholder data. Send the logs to the centralized log server.

  5. Protect the database access credentials used by applications. Allow the access credentials used by the database application to be used by that application only. Do not allow these credentials to be used by users or other processes. This is best accomplished by web applications. If you use a web application, you can configure the database server to use a white list. A white list is a list of hosts that are exclusively allowed to access some functionality while rejecting all access that does not originate form the web application server. On UNIX-style systems, white lists are usually configured by editing the /etc/hosts.allow and /etc/hosts.deny files.
