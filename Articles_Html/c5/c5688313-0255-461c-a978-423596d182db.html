Applies To

Applications written using Scala and Play Framework 2.1.0+

What to Check For

Ensure that all input validation is performed on the server. Client-side validation should only be used for performance and usability reasons.

Why

Client-side validation is executed by the client and can be easily bypassed. Client side validation is a major design problem when it appears in web applications. It places trust in the browser, an entity that should never be trusted.

How to Check

Use the following steps to establish a validation strategy:

  1. Identify all sources of input. An application can take input via various ways such as a web interface, database, file system or other software running on the server. During design, identify all potential sources of input to your application. Scour the source code to discover sources of input that may have been missed in the design. The most common sources are:

    • URL based parameters
    • Form based parameters
    • Hidden fields
    • Cookies
    • HTTP headers (Host, accept types, www authentication, cache settings, encodings, etc)
    • Local filesystem
    • Database
    • Other services running on the system
    • JavaScript variables
    • File upload and attributes (filename, size, data, etc)
    • DNS results or host names
    • External component call return values (AJAX, etc)

  2. Identify validation routines. Each input source should have a server-side validation routine associated with it. Ideally the validation will occur as soon as the input reaches your application. Shared validation routines are better than creating many throughout the code base and may be located in a single component.

    Validation may be identified by any of the following:

    • Custom Scala Validator class
    • Regular expression use (scala.util.matching.regex)
    • Method names that contain validate or validator
    • Client-side validation is often linked to events:
      • onSubmit
      • onChange
      • onFocus
      • onBlur
      • onKeyDown/Press/Up

  3. Ensure validators run on the server. Validation routines should run on the server. Server-side validation guarantees that malformed input will be caught and discarded regardless of the state that the client is in. As an optional feature, validation can also be implemented client side in order to improve application performance and user experience.

    Server-side validation is done within the application code. Ideally, validation occurs immediately after canonicalization, and canonicalization should be done as soon as the input reaches your application. In web applications, look for application-supplied validators, such as a Custom Validator, to validate and constrain input after it is received from an input source, such as a Form parameter. Check for the use of regular expressions and scala.util.matching.regex, or the use of a validator that checks numeric ranges.

How to Fix

Client-side validation should only be used to improve user experience, never for security purposes. Client-side validation can improve application performance by catching malformed input on the client, thus saving a roundtrip to the server. However, client-side validation can be easily bypassed, and places trust in the browser, an entity that should never be trusted. Always use server-side validation to protect your application from malicious attacks.

Use the following steps when developing validation for client-side input:

  1. Never trust the browser. Because the browser is running on the user's machine, it can be fully controlled by the user. Therefore, any client-side validation code can be controlled and bypassed by an attacker.

  2. Use JavaScript only to enhance your pages, but never for security. JavaScript is useful for enhancing your application's presentation. However, it has no mechanism to protect the integrity of its code. Do not rely on JavaScript to enforce security decisions.

  3. Place a server-side validator for all input. Execute validation on the server side. Use the following steps to properly set the validation routines:

    • Identify input. Determine the data that is to be passed from the client to your application. Input can take the following forms on the client-side: 

      • URL-based parameters
      • Form-based parameters
      • Hidden fields
      • Cookies

    • Understand the input. Understand the use and constraints associated with each input. Be clear how each input is used. Define the format and type of each input. Build a server-side validator for each format and type of expected input.

    • Create a set of validators. Build a validator for each type of input. Place the set of validators on the server-side of your application. If there are validators on the client-side, make sure the same validators are implemented on the server-side as well.

    Once developed, it is important to place the validation routines at your application's trust boundary. Because your application's trust boundary extends only to your application's process space, do not consider the user's browser to be within your application's trust boundary.
