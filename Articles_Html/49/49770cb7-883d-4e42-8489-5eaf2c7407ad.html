##What to Check For##

Verify that risks for the applications in the cloud have been identified.

##Why##

By identifying application security risks, you gather details about the impact to an organization when a threat scenario is realized. If the risks are not enumerated and measured, there is great difficulty in having sufficient resources ready for an incident response.

When risks are listed in your security framework, it is clear how immediate your response needs to be when attackers overcome established controls.

NIST SP800-137 requires risk assessment (RA-03) and categorization of risk (RA-02). It is also important to measure the magnitude of harm that might result from unauthorized access of an organization's resources.

##When##

When running an application in the cloud, you are required to secure the application. A variety of infrastructure safeguards are provided by AWS. You must always provide additional protection, especially when your application works with sensitive information. Identifying application security risk helps guide the assignment of resources and the actions of your incident response team.

##How to Check##

To make sure that risks for the applications in the cloud have been identified:

  1. Identify responsible personnel. Find and interview the employees that are responsible for identifying risks to the applications in the cloud.
  2. Examine documented risks. Review the documentation that includes the risks that have been identified and assess its quality.

##How to Fix##

The following activity provides a suggested method for identifying and assessing risks for a cloud application. The goal is to maintain a secure posture by allocating resources for the protection of your cloud assets. The products include a security frame with categories, vulnerabilities, controls and risks laid out for further analysis.

  1. Choose a security framework. Select a security framework so that you can organize the application's vulnerabilities and threats. If you do not already have a security framework, see https://msdn.microsoft.com/en-us/library/ff649461.aspx.

  2. Decide on classification categories. Select common categories (input and data validation, sensitive data, and cryptography, for example) to help segment vulnerabilities into areas of concern.

  3. Examine vulnerabilities. In your security frame add vulnerabilities to each category. For example, the cryptography category has vulnerability to failure to secure keys, as well as insecure distribution of keys. For more information on identifying vulnerabilities, see https://nvd.nist.gov/.

  4. Use a threat model. Use a threat risk modeling tool or method to survey the web application, decompose it, and identify vulnerabilities and threats. This will minimize the use of resources, and provide an efficient method to be used for identifying attack vectors. For more information about threat risk modeling, see https://www.owasp.org.

  5. Identify attack vectors. Use the results of threat modeling to identify specific attack vectors for the application. Attack vectors may be associated with multiple threats. Once identified and integrated with your security frame, you can examine the controls and incidence response items which apply.

  6. Identify threat scenarios. For each attack vector, develop an impact statement which lists the possible scenarios as well as actor, motive, access mode, and outcome.

  7. Rate each scenario. Assign an impact rating to each scenario. For example, use values such as: none (0); low (1); medium (3); or high (4).

  8. Identify risks. Each risk consists of an impact statement, and a probability, which is calculated from the threat outcome and the consequence. The impact statement names the resources that will become unavailable or lost as the scenario plays out if unchecked. For more information about identifying and assessing risks, see http://resources.sei.cmu.edu/.
