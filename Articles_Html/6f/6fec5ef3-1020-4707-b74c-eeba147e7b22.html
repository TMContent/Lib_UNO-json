What to Do

When using SQL Server authentication, choose a strong password.

Why

Using strong passwords decreases the likelihood that an attacker is able to gain access to an account by guessing the password. Failing to use sufficiently strong passwords might make it possible for an attacker to succesfully brute-force crack a password without resorting to millions of attempts. 

When

This guideline is applicable whenever SQL authentication is used. Never use weak passwords for SQL authentication.

How

Here are the steps that should be followed to create a strong password:

  1. DO use a password with mixed-case letters. Use uppercase letters throughout the password.
  2. DO NOT just capitalize the first letter, but add uppercase letters throughout the password.  
  3. DO NOT use a network login ID in any form (reversed, capitalized, or doubled as a password).
  4. DO use a password that contains alphanumeric characters and include punctuation, such as ! and $.  
  5. DO NOT use your first, middle or last name or anyone else's in any form. Do not use initials or any nicknames.  
  6. DO NOT use a word contained in English or foreign dictionaries, spelling lists, or other word lists and abbreviations.  
  7. DO use at least seven characters.  
  8. DO NOT use other information easily obtainable. This includes pet names, license plate numbers, telephone numbers, identification numbers, the brand of your automobile, the name of the street you live on, and so on. Such passwords are very easily guessed by someone who knows the user.  
  9. DO use a seemingly random selection of letters and numbers.  
  10. DO NOT use a password of all numbers, or a password composed of alphabet characters. Mix numbers and letters.  
  11. DO use a password that can be typed quickly, without having to look at the keyboard. This makes it harder for someone to steal a password by looking at the keyboard (also known as "shoulder surfing").  
  12. DO NOT use dates e.g., September, SEPT1999 or any combination thereof. 
  13. DO change passwords regularly. The more critical an account to network integrity (such as root on a Unix host or Administrator on Windows NT), the more frequently the password should be changed. This change stops someone who has already compromised an account from continued access.
  14. DO NOT use keyboard sequences, e.g., qwerty.  
  15. DO NOT use a sample password, no matter how good, from a book that discusses information and computer security.  
  16. DO NOT use any of the above things spelled backwards, or in caps, or otherwise disguised.   
  17. DO NOT reveal a password to anyone, as this automatically invalidates it's strength. 

Problem Example


An ASP.NET application uses SQL server for it's persistence layer. There is an instance of SQL injection in the application, and the SQL administrative account 'sa' password is '1234'. The database account that the application uses is configured with minimal privileges. Here's how a hacker can run arbitrary commands in SQL:

1. The hacker locates an instance of SQL injection on the Web site.

2. The hacker determines that he needs to run the command under the sa account, since the database account that the application uses lacks the appropriate privilege to carry out the atttack that the hacker has in mind.

3. The hacker knows he can run a SQL command with an alternate identity if he runs an OPENROWSET command such as:

      SELECT * FROM OPENROWSET 
        ('SQLOLEDB',';'sa'; 
        '<password>','<query to be 
        executed>')

4. The hacker comes up with a SQL injection attack that will let him pass in a valid query. He uses sa as the username and guesses the password.

5. The application errors out, which lets him know that the password was invalid.

6. The hacker creates a simple brute-force program to crack the sa password. since the password is '1234', it will most likely be discovered with minimal effort.

7. Once the hacker has the sa password, he can continue to execute the SQL injection attack using the OPENROWSET command.

Solution Example

An ASP.NET application uses SQL server for it's persistence layer. The SQL administrative account 'sa' password is set to  'rYK56lm4$'. The database account that the application uses is configured with minimal privileges. This is a better password choice then using a password like '1234' for several reasons:

  • uses mixed case letters
  • uses special characters beyond alpha numeric
  • does not look like any default passwords that a system might use
  • uses several characters that are close to the home row on a QWERTY keyboard, which makes it much faster to tyep and harder to shoulder surf.
  • does not correspond to any words in language dictionaries

Note: DO NOT use any of the above examples directly as password choices! They are immediately invalidated as safe since they are included in a security reference guide that is publically available.
