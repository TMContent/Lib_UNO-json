Applies to

  • PCI DSS Requirement 6.4

What to Do

Test and document all changes, including updates, made to software code and applications.

Why

Testing helps ensure that patches do not introduce new vulnerabilities or break functionality. Changes should be documented to help find the cause of a vulnerability if one is introduced by the changes.

When

All changes to software code, including patches, should be tested and documented before being deployed.

How

Perform the following actions to implement change control procedures in compliance with PCI DSS:

  1. Test all changes. Do not use production data for testing, especially live PANs. Test all changes, including updates and significant configuration changes. Set up a test environment to closely simulate the production environment. Test all changes for the following:
    • Validation of all input
    • Validation of error handling
    • Validation of secure cryptographic storage
    • Validation of secure communications
    • Validation of proper role-based access control

  2. Define change-control procedures. PCI DSS requires following change control procedures for all changes to system components. These procedures may be stated explicitly in a separate document or included in the information security policy. The procedures are the ones described in this article and include documentation of impact, management sign-off, functionality testing and back-out procedures.

  3. Designate a testing environment. Design a simulated&nbsp;test environment that uses hardware and software similar to that used in the production environment. If possible, set up the test environment like a scaled-down version of the production environment. Identify the components of the production environment that will be affected by the change being tested. Install and configure hardware and software similar to the affected components in the test environment. Install and review changes in the test environment before deploying them in the production environment. Staff working in the production environment should not be working in the test environment

  4. Include customer impact considerations. While testing changes, consider and document the possible customer impact of each change. Things to consider are the effect the change would have on the user experience and the functionality that may be added or removed. Attach the customer impact considerations to the change-control documentation.

  5. Require a management sign-off. The management should sign off on each change before it is deployed. The signed documents should be included in the change-control documentation.

  6. Document the test results. The objective of testing is to make sure that the change to the code produces only the desirable behavior. The results of how well the code changes have achieved these objectives&nbsp;should be noted. These results should be included in the change-control documentation.

  7. Define back-out procedures. Design and document back-out procedures for each change. Include the descriptions of the back-out procedures in the change-control documentation. Typically, the back-out procedures should describe how to uninstall the patch or how to reinstall the earlier, stable version of the software.
