Applies To

  • iOS applications that send or receive sensitive data over the network.

What to Check For

Verify that self-signed TLS certificates are disabled.

Why

Allowing the use of self-signed TLS certificates enables some types of man-in-the-middle attacks.

How to Check

To verify that self-signed TLS certificates are disabled:

  1. Search for code that allows self-signed TLS certificates. Search application code for the presence of the continueWithoutCredentialForAuthenticationChallenge selector - the presence of this selector indicates that self-signed TLS certificates may be allowed.
  2. Verify that self-signed TLS certificates are not allowed in production releases. If self-signed TLS certificates are allowed for development purposes, make sure that they are disabled in production releases. The recommended way to disable development code in production releases is through #define and #ifdef compiler directives.

How to Fix

Self-signed TLS certificates are not allowed by default. Do not allow them. Avoid using the allowsAnyHTTPSCertificateForHost method in production releases, because it allows self-signed TLS certificates. Do not use the continueWithoutCredentialForAuthenticationChallenge selector within the didReceiveAuthenticationChallenge delegate method of NSURLConnection objects, because it enables self-signed TLS certificates.

Communication Security Guidelines

  • Disable Self-signed TLS Certificates
  • Use TLS to Protect Communications
  • Use Strong TLS Configurations

Communication Security Checklists

  • Self-signed TLS Certificates Are Disabled
  • TLS Is Used to Protect Communications
  • Strong TLS Configurations Are Used
