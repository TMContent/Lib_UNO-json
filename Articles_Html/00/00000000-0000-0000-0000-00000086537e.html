What to Check For

Ensure that data coming from WebSockets is validated.

Why

Validate data coming from WebSockets in order to make it more difficult for an attacker to send malicious data to the application. An attacker might be able to take over the remote end of a WebSocket and attempt to leverage this to control the user's browser. Validating data received from WebSockets severely limits the ability of attackers to attempt to exploit the client-side part of the application.

How to Check

To verify that data coming from WebSockets is validated:

  1. Identify code that receives data from WebSockets. Search application code for all references to the onmessage event. Create a spreadsheet to list all code that receives data from WebSockets.
  2. Identify data formats. For each piece of code that receives data from WebSockets, define valid data formats that are expected by the application. For example, a piece of code might be expecting a time, a name, or a social security number, and each of these data types has a distinct valid format. Add information to the spreadsheet about these expected data formats. Include examples and be as specific as possible.
  3. Identify validators. For each distinct data type, make sure that a validator is present. Review each vaildator function to make sure that it checks the length, characters, format, and numerical range, as applicable.
  4. Verify that data coming from WebSockets is validated. For each piece of code that receives data from WebSockets listed in the spreadsheet, verify that validator functions are called before data is processed. Data should only be processed if it passed through the corresponding validator function and was found to be valid.

How to Fix

To validate data coming from WebSockets:

  1. Identify code that receives data from WebSockets. Search the application code for all references to the onmessage event. Create a spreadsheet to list all code that receives data from WebSockets.
  2. Identify data formats. For each piece of code that receives data from WebSockets, define valid data formats that are expected by the application. For example, a piece of code might be expecting a time, a name, or a social security number, and each of these data types has a distinct valid format. Add information to the spreadsheet about these expected data formats. Include examples and be as specific as possible.
  3. Define validators. For each distinct data type, write a validator function that determines whether a piece of data matches the correct format or not. Regular expressions are usually used for this task. Validate data length first, before using regular expressions, to prevent regular expression denial of service attacks. The general pattern for validating data formats is: validata data length, validate valid characters, validate patterns of structured data (like social security numbers or dates), and validate ranges of numerical data.
  4. Validate data. For each piece of code that receives data from WebSockets listed in the spreadsheet, add code to validate the data immediately after receiving it. Use the validator that matches the expected data type. Continue to process the data only if it has the correct format.

WebSocket Guidelines

  • Validate Data Coming from WebSockets
  • Check the Origin Header in the WebSocket Handshake
  • Use TLS to Encrypt WebSocket Connections
  • Check the Origin Attribute of WebSocket Messages
  • Use the Latest WebSocket RFC 6455 Standard
  • Do Not Tunnel TCP via WebSockets
