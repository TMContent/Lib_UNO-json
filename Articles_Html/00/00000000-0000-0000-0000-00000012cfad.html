Applies To

  • ASP.NET

What to Check For

Make sure that reflection is avoided as much as possible.

Why

Reflection allows executing data as code. If user-controlled data is executed as code, an attacker may be able to execute arbitrary code to take over the application.

How to Check

To make sure that reflection is used as little as possible:

  1. Find all code that uses reflection. Search the code for the use of System.Reflection namespace to find code that uses reflection.
  2. Determine if reflection is necessary. Examine each case where reflection is used and determine whether it is possible to do the same thing without using reflection or to remove the code that uses reflection altogether.

How to Fix

To avoid using reflection:

  1. Find all code that uses reflection. Search the code for the use of System.Reflection namespace to find code that uses reflection.
  2. Remove code that uses reflection if possible. Consider whether the code that uses reflection is absolutely necessary. Think of ways to replace the reflection code with something else that does the same thing.
  3. Don't pass user data to code that uses reflection. Review code that uses reflection and make sure that it doesn't accept user data. The nature of reflection is that it allows passing code as data, so there isn't really a safe way to handle user input passed to reflection.
  4. Validate user data passed to code that uses reflection. If it's absolutely necessary to use reflection with user data, then data passed to it must be rigorously validated. Use positive input validation to check the user supplied data. Ensure that only valid input is accepted. Limit user data to as small data set as possible. It is a good idea to include a highly visible warning in the application's documentation and code comments, precisely stating the conditions under which reflection with user data is necessary.
