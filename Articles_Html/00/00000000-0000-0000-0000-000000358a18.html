Applies To

  • HTML5

What to Check For

Verify that different applications are hosted on different subdomains.

Why

Different applications should be deployed on different subdomains to prevent multiple applications from sharing the same localStorage object. This will mitigate the risk of unauthorized disclosure. Applications deployed on the same domain share the same localStorage object. If one application is compromised, the data stored in localStorage by the other applications on the same domain will also be compromised.

How to Check

To verify that different applications are hosted on different subdomains:

  1. Enumerate the applications. Create a spreadsheet to list each application and the domain/subdomain that each application is hosted on.
  2. Check the locations of the applications. Examine the spreadsheet to verify that no two applications run on the same domain/subdomain.

How to Fix

To host different applications on different subdomains:

  1. Identify all the applications on the same domain. Create a spreadsheet of all the applications on the same domain.
  2. Create a separate subdomain for each application. For each application in the spreadsheet, use whatever mechanism is used to manage domains to create subdomains. In practice, you might need to ask the person who manages the domains to do this. Ideally, each subdomain should be isolated from the others in a manner that prevents file-system access between each application. This can usually be done as follows: For each application, create an OS user and provide only this OS user with access to the application's web root folder.
  3. Move each application to its separate subdomain. Copy each application's files to its new subdomain. Make sure each application works properly at its new location. It's probably best not to delete the files on the original domain, but to move them to a location outside of the web root. Moving the original files outside of the web root makes them inaccessible from the Internet, but makes it easy to restore them if necessary.
  4. (Optional) Create redirects from the old URLs to the new subdomains. Use a redirection mechanism, such as JavaScript redirection, a server setting, or a script that sends an HTTP redirection header, to send users from each application's original location to its new location. Do this for each application that has been moved.
  5. (Optional) Notify the users of any changes that were made. Depending on how critical the data handled by the application is, it might be prudent to notify the users that a change in the URL has taken place, so that they are not alarmed when they notice this change. A simple way to provide this notification is to put a note on the front page of the application. Another way is to send e-mail to the users of the application to explain the change.

Local Storage Guidelines

  • Host Different Applications on Different Subdomains
  • Do Not Store Sensitive Information in Local Storage
  • Use Session Storage Instead of Local Storage If Possible
  • Do Not Store Session Identifiers in Local Storage
  • Do Not Store Authentication Information in Local Storage

Local Storage Checklist Items

  • Different Applications Are Hosted on Different Subdomains
  • Sensitive Information Is Not Stored in Local Storage
  • SessionStorage Is Used Instead of LocalStorage If Possible
  • Session Identifiers Are Not Stored in Local Storage
  • Authentication Information Is Not Stored in Local Storage
