What to Do

Avoid dangerous APIs.

Why

Some of the most dangerous vulnerabilities are usually caused by the same few APIs on most platforms. Each platform often has its own peculiar features that are prone to causing security vulnerabilities when not used with great care. There are usually safer, alternative APIs available that can do the same thing. It is usually simpler to implement the application using safe APIs than to have to maintain potentially dangerous code. Some notable examples of dangerous APIs are eval(), system(), printf(), etc. but each platform has its own issues and APIs to avoid.

How

To avoid dangerous APIs:

  1. Identify the dangerous APIs for your platform. Look up information about dangerous APIs for your platform. There are additional articles on this subject in TeamMentor. Make a list of each dangerous API for your platform.

  2. Search code for dangerous APIs. Use the list of dangerous APIs as a list of search keywords to search code for potential trouble area.

  3. Rewrite code to use safe APIs. Wherever calls to dangerous APIs are found, rewrite the code to use safe API alternatives.

  4. Document code that uses dangerous APIs. Use this documentation when patching discovered vulnerabilities - when searching for code responsible for vulnerabilities, checking code that is known to use dangerous APIs is a good idea.
