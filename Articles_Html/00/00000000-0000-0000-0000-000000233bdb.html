Applies To

  • HTML5

What to Do

Use session storage instead of local storage as much as possible.

Why

Session storage should be used instead of local storage as much as possible, because session storage only stores data for one session, and, in the event of a compromise, only the data from the latest session would be available to the attacker. This reduces the likelihood that large amounts of information could be leaked. Data stored in local storage, on the other hand, does not expire unless it is explicitly cleared, which might lead to a large amount of information accumulating in local storage. If a compromise occurs, all data in local storage might become available to the attacker.

How

To use session storage instead of local storage:

  1. Find all code that uses local storage. Search the application code for references to the localStorage object, to locate all code that stores or accesses data in local storage. Create a spreadsheet to list all code that stores or accesses data in local storage.
  2. Identify all data types stored in local storage. Examine each piece of code that stores or accesses data in local storage. Create a spreadsheet of each distinct data type that is stored in local storage, such as first name, last name, date of birth.
  3. Identify whether session storage can be used instead. For each data type that is stored in local storage, consider whether that data needs to persist between sessions. If a data type does not need to persist between sessions, it should be stored in session storage instead of local storage. In the spreadsheet of distinct data types stored in local storage, note which data types should be stored in session storage instead.
  4. Rewrite code that uses local storage to use session storage. Using your spreadsheets and notes, identify code that stores data in local storage which should be stored in session storage. Rewrite each piece of code to use session storage instead of local storage.

Local Storage Guidelines

  • Host Different Applications on Different Subdomains
  • Do Not Store Sensitive Information in Local Storage
  • Use Session Storage Instead of Local Storage If Possible
  • Do Not Store Session Identifiers in Local Storage
  • Do Not Store Authentication Information in Local Storage

Local Storage Checklist Items

  • Different Applications Are Hosted on Different Subdomains
  • Sensitive Information Is Not Stored in Local Storage
  • SessionStorage Is Used Instead of LocalStorage If Possible
  • Session Identifiers Are Not Stored in Local Storage
  • Authentication Information Is Not Stored in Local Storage
