Applies To

  • HTML5

What to Check For

Verify that user permission is required before sending the manifest file.

Why

The user should be informed that the application is being cached to mitigate the risk of cache poisoning attacks.

How to Check

To verify that user permission is required before sending the manifest file:

  1. Identify whether the application is likely to be cached. An accurate manifest file is necessary for an HTML5 application to be cached properly. If your application has a manifest file, the application is likely to be cached (presumably the developers have decided that the user might want to cache the application and use it off-line). If there is no manifest file, consider whether the application is likely to be cached and used off-line. If so, write a manifest file for it.
  2. Verify that the user is asked for permission before sending the manifest file. Log in to the application and attempt to cache it. Take note of whether the application warns you about untrusted networks before it gets cached.

How to Fix

To notify the user about caching the application:

  1. Identify whether the application is likely to be cached. An accurate manifest file is necessary for an HTML5 application to be cached properly. If your application has a manifest file, the application is likely to be cached (presumably the developers have decided that the user might want to cache the application and use it off-line). If there is no manifest file, consider whether the application is likely to be cached and used off-line. If so, write a manifest file for it.
  2. Write a message to inform the user about the risks of caching the application. Write a message that explains to the user that the application might be substituted by a malicious one if the user is connected to an untrusted network.
  3. Ask the user for permission before sending the manifest file. Use the following steps:
    1. Find the page that references the manifest file in the html tag (let's call this the manifest-loading page).
    2. Next, find all the pages that send the user to the manifest-loading page. On all such pages, before you send the user to the manifest-loading page, display the warning message which you created in the previous step. Note that referencing the manifest file in the html tag causes the browser to download the manifest file and possibly cache the application. The user should be warned before this happens.
    3. If it is impractical to place warnings on all the possible pages that lead to the manifest-loading page, it might be simpler to place a warning directly on the manifest-loading page (let's call this the warning page). For this to work, you'll need to remove the manifest reference from the warning page, because otherwise it will be loaded regardless of the user's response to the message. Create another page to load the manifest (let's call this the caching page). Remove the reference to the manifest from the warning page. On the warning page, place a link to the caching page, for use when the user has accepted the warning and decides to cache the application anyway.

Privacy and Trust Guidelines

  • Ask User Permission Before Using Geopositioning
  • Ask User Permission Before Sending a Manifest
  • Tell Users to Clear the Cache after Browsing Through Untrusted Networks

Privacy and Trust Checklist Items
