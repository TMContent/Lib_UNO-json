What to Check For

Verify that logs are sent to a centralized log server.

Why

Sending logs to a centralized log server makes it simpler to secure and monitor logs on a network with multiple log-generating devices.

How to Check

Perform the following actions to verify that functionality to send logs to a remote server is implemented:

  1. Examine documentation. If remote logging is implemented, it should be documented. Review documentation for information about remote logging functionality.

  2. Identify the logging subsystem. The logging subsystem should provide an API that allows generating log entries from application code. If this API or the logging subsystem don't exist, write them. Make a list of functions that may be used to generate log entries.

  3. Review the logging subsystem. Use the list of logging functions and trace them to determine where the log entries end up. Verify the code exists that sends log entries to a remote log server.

How to Fix

Use the following steps to implement sending logs to a centralized log server:

  1. Design the logging subsystem. Design and implement a class or library specifically for logging. When designing the logging subsystem, consider sending logs to a remote log server. Consider the following factors when designing the logging subsystem:

    • Log throttling
    • Logging important event details
    • Logging important security operations
    • Logging important business operations
    • Monitoring the use of privileged accounts
    • Logging authentication events
    • Avoiding including sensitive information in logs
    • Logging unusual activity

  2. Choose a protocol for communicating with the log server. The Internet standard for sending logs remotely is syslog (RFC 5424). It is recommended to use syslog, because it is supported by most centralized logging solutions.

  3. Implement the logging subsystem. Write code that generates log entries and sends them to a remote log server via your protocol of choice.
