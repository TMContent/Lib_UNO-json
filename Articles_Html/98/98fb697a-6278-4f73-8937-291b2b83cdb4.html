What to Check For

Ensure that all error messages are generic and do not disclose any valuable or sensitive information that could be useful to an attacker.

Why

Attackers often probe web applications for error messages as a first attempt to understand the underlying technologies in place. An attacker will look for discrepancies between error messages to discover when different code paths are being taken or to discover information about the quality of their input. Error messages that are directly echoed from backend systems such as SQL or OS error messages can help a malicious user choose what to input&nbsp;for&nbsp;their next attack.

How to Check

For every error message check that:

  1. The error message is simple and generic. In particular do not include extra information regarding:
    • The quality of the information being processed
    • The mechanisms processing the data
  2. Debug information is not echoed to the user.
  3. There is only one error message displayed on all authentication failures. This message should be basic and not contain any information that can be used to discern between the following types of failures:
    • Invalid username
    • Valid username, but invalid password
    • Account locked
  4. A custom error message is used instead of the default error message which describes the error in detail:
    • Set customErrors to RemoteOnly or On in the web.config file

For dynamically generated error messages also ensure that:

  1. Use of dynamic error messages is minimized.
  2. Error messages that report the same message are exactly the same.

Problem Example

The following code authenticates a user. Unfortunately, the code reveals through its error messages whether the credentials sent represent a valid user or an invalid user:

public void login()
{
      if (!validUser())
      {
            throw new Exception("User not found.");
      }
      if (!validPass())
      {
            throw new Exception("Wrong password.");
      }
      commit();
}
