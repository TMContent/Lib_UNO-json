Applies To

  • HTML5
  • Server-side

What to Do

Force the user to re-authenticate when executing security-critical functionality including, but not limited to, change of password, account modifications and critical transactions.

Why

Forcing the user to re-authenticate verifies the user's decision to execute the given functionality and preserves the application's integrity in the event that a user's account has been compromised via session hijacking.

How

Follow these steps when forcing re-authentication:

  1. Identify critical functionality. Evaluate your application's design and requirements to determine which parts and resources of your application are very important to your organization's business model or your application's security architecture. This could include activities such as financial transactions or resetting user passwords.
  2. Adopt a re-authentication mechanism. Establish a mechanism that requires the user to verify their authenticity:
    1. Current password. The user uses their current password to confirm their identity. This technique ensures that the user is aware of the action to be performed and holds responsibility for executing it.
    2. CAPTCHA. CAPTCHA are mechanisms designed to ensure that a human, not an automated script, is using the system. A CAPTCHA should be used in addition to, not instead of, a password. Many CAPTCHA systems embed distorted letters or numbers into displayed images, and require the user to enter the letters or numbers. There are a number of problems with many CAPTCHA systems, so don't rely on them too heavily. Many graphical CAPTCHAs can be trivially broken by optical character recognition systems, so it is difficult to ensure that a CAPTCHA is sufficiently strong. If CAPTCHAs are reused, the attacker can simply record the session IDs of CAPTCHAs with known values and replay them. CAPTCHAs can also be broken by using a man-in-the-middle technique where the image is saved and forwarded to another site controlled by the attacker. An unsuspecting user of that site solves the CAPTCHA and the attacker forwards the answer to the victim site. CAPTCHAs also create accessibility problems for users, as blind and sometimes even color-blind individuals can find them impossible. If you choose to implement a CAPTCHA, be aware of the trade-offs involved and ensure that the system you use is appropriately protected against any anticipated attacks.
    3. Second factor. If your application supports it, then a second factor of authentication, such as a token, could play a role in the re-authentication scheme.
  3. Enforce the re-authentication mechanism. Once you've identified the critical functionality and established the re-authentication mechanism, enforce the re-authentication mechanism before permitting any security-critical actions.

Essential Server-side Security Guidelines

  • Validate All Input
  • Use Prepared Statements for SQL Queries
  • Encode Input Included in Output
  • Force Reauthentication when Executing Critical Functionality
  • Include Unique Tokens in HTTP Requests
  • Use Role-based Authorization
  • URL-Encode Data in HTTP Response Headers

Essential Server-side Security Checklist Items

  • All Input Is Validated
  • Prepared Statements Are Used for SQL Queries
  • Input Included in Output Is Encoded
  • Reauthentication Is Enforced When Executing Critical Functionality
  • Unique Tokens Are Included in HTTP Requests
  • Role-based Authorization Is Used
  • Data in HTTP Response Headers Is URL-Encoded
