Applies To

  • Amazon Web Services (AWS)

What to Do

Follow a security assessment plan (SAP) to initiate, execute, and communicate the results of security testing for cloud deployment. Identify security architecture, instances and applications that will be tested for security function. Select methods, systems and tools and systems that will be used to perform the testing. Coordinate with the cloud service provider (CSP) to authorize testing.

Why

Security testing for cloud deployments ensures the security posture is maintained at a safe level. Testing is required since the CSP provides protection from threats at lower levels, while the account holder must use additional measures to safeguard their applications and stores.

When periodic security testing is performed, new findings are beneficial to the organization. Testing may reveal that controls in place are not 100% effective against newly discovered vulnerabilities in the deployment.

Security testing also enables sharing of security best practices between the CSP, organization, and third party assessor organization (3PAO). This can help prevent future security issues.

When systems are updated and/or re-configured, affected sub-components, processes, and custom software must be tested to ensure security controls are effective.

When

New vulnerabilities are being discovered continually by malicious individuals and researchers. Your security assessment plan should include details about the period and frequency for testing of controls.

How

Security testing is part of a security assessment plan, which can be broken down into these steps:

  1. Scope the security testing. Name the system that that is slated for testing and include the regions for components that will be tested. Describe the system and list IP addresses which will be affected. This can be obtained from the cloud service provider (CSP). IP addresses can be listed by network ranges and CIDR blocks. Insert URLs and the associated login IDs that will be used for testing. Also record the function of the resource under test. Also include databases with the same information. List roles, functions, and restrictions that will be tested.

  2. State the testing assumptions. These are default assumptions that are edited for each engagement. The CSP will provide resources such as documentation and administrators with system and infrastructure knowledge. If necessary, the CSP will provide credentials to authenticate to protective devices. The CSP should provide network access to vulnerability management services so that testing authority can see results. Upgrades and patches will not be done during testing.

  3. Prepare testing methods. Authorities, such as FedRAMP, can provide documented test methods and descriptions for testing security controls. Your security engineer can edit the tests as necessary to accommodate business rules.

  4. Prepare test plan. List the members of the test team and the role each member will play. Include basic contact information. Also list the CSP points of contact, when available, phone, etc. Describe any test tools to be used, including version, developer, and function. Include manual methods you may use, such as manual SQL injection. Include a simplified schedule for the overall plan and test milestones.

  5. Disclose rules of engagement. Rules of engagement (ROE) describes the notifications and disclosures between the account owner, 3PAO, and CSP. The ROE lists roles, test methods, functions to perform, and contact information if not supplied already.

  6. Conduct security testing. Security testing is initiated according to the schedule. All terms and conditions agreed to must be followed. This will minimize the risk exposure of tested systems. The following tests are typical for cloud deployments:

  • Port scans and network queries
  • Network sniffing, traffic monitoring, traffic analysis
  • Login attempt with account names with weak or guessed passwords
  • SQL injection and input parameter testing
  • Use of exploit code
  • Password cracking
  • Spoofing network traffic
  • Altering system configuration
  • Adding user accounts.
  1. Compile and communicate security test results. When testing has ended, the results are given to representative of the CSP, 3PAO, and organization's security engineer. Reports are sent out the prescribed number of days after testing has ended.
