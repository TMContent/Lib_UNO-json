What to Do

Set the customErrors mode attribute to On to prevent detailed error messages from reaching the caller.

Why

Detailed error messages provide an attacker with valuable system&nbsp;configuration information, making the system far easier to compromise.&nbsp; Furthermore, some detailed error messages may contain sensitive user information.&nbsp; Depending on how your application handles errors, leaving detailed error messages on could directly result in the disclosure of sensitive customer data.

When

All applications should set customErrors to On for normal operations.&nbsp; RemoteOnly is an acceptable choice for situations when local debugging is required in an emergency, but is not the preferred setting.

How

Use the following steps to prevent detailed error messages:

  1. Set CustomErrors. Prevent detailed error messages from being displayed by setting the mode attribute of the <customErrors> element to On. This will ensure that&nbsp;callers receive only filtered exception information.&nbsp; Do not use mode="Off" because this allows detailed error pages intended for application developers that contain system-level information to be returned to the client.

  2. Choose a default error page. Use the <customErrors> section of the Web.config file as shown in the following code example to specify a default error page to display, along with other required error pages for specific HTTP response codes that indicate errors.

    The following example shows <customErrors> set correctly, with custom error pages specified:

    <customErrors mode="On" defaultRedirect="ErrDefault.aspx">    
    &#9;<error statusCode="401" redirect="ErrUnauthorized.aspx" />    
    &#9;<error statusCode="404" redirect="ErrPageNotFound.aspx" />   
    &#9;<error statusCode="500" redirect="ErrServer.htm" />
    </customErrors>

    The defaultRedirect attribute allows you to use a custom error page for your application, which, for example, might include support contact details.&nbsp; Use these application-wide error pages to provide user-friendly responses for errors that are not caught in a structured event handling.
