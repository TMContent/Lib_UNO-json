What to Do

Prepare an incident response plan.

How

Formulate the incident response plan and develop an overall incident response capability in compliance with PCI DSS by performing the following tasks:

  1. Organize the incident response team. Define roles of the incident response team members. Define responsibilities of team members. Organize the incident response team to monitor the centralized log center 24/7 and to have the capability to react to suspected compromises. This means the team must be organized in shifts so that at least one person is monitoring the log server at any time. Make sure the rest of the team is easy to reach so that the staff who is monitoring can summon others quickly. Define communication strategies to be used by team members to talk to each other and for other employees to contact the incident response team. The communication strategies may involve cell phones, email, instant messaging, etc. Consider communications security and do not disclose sensitive information over public networks. Define procedures and delegate responsibilities for notifying payment brands and customers in the event of a suspected compromise. It may not always be required to notify the customers, but it is good to have procedures for this in place in advance.

  2. Define incident response procedures. Define specific incident response procedures. These procedures should reflect your computing environment and involve the technologies you use. A good outline to define these procedures is:

    • Place the entire company on full alert and isolate the system component, which is considered to be compromised.
    • The incident response team should investigate the quarantined component and examine the logs.
    • If malicious software is detected or there are signs of unauthorized changes to the system, payment brands and possibly customers should be notified.
    • The investigation may need to be escalated to use external resources, such as federal investigators or investigators sent by the payment brands. Predicting the course of a large-scale investigation is difficult, especially when external parties that are not under the command of the company are involved, therefore this stage of incident response does not need to be explicitly defined, but it may be helpful to mention the capability for this course of action.

  3. Define recovery and continuity procedures. These procedures should be specific to your environment and technologies, but they should include the following:

    • Fix the vulnerability that made the compromise possible before resuming business operations.
    • In the event that some component part is compromised, such as wireless for example, it may be possible to shut off the affected component and carry on business operations. In the event of a large-scale compromise this may not be an option.
    • The recovery and continuity plans should include worst-case scenarios.
    • One method for ensuring continuity is to have a small-scale duplicate infrastructure that includes additional security features available for use while the main system is investigated. A smaller scale infrastructure is easier to secure - it should not be merely a clone of the main system, but a significantly hardened system that includes only the minimum features to assure business continuity.

  4. Test the incident response plan. Test the incident response plan when performing the annual penetration test. Include this fact when documenting the penetration test.

  5. Conduct training for the incident response team. Enroll the members of the incident response team in an information security-training program. Security Innovation offers information security training services.

  6. Monitor alerts. Configure the log server or the IDS/IPS to send alerts to the members of the incident response team. Since there should be always at least one member of the incident response team watching the log server, at least one person should be always available to respond to alerts. Give the members of the incident response team each other's contact information and contact information of other people and organizations that may be helpful, such as the local police, the insurance company, information security professionals, etc.

  7. Update documentation and policy. Update the incident response plan when fixing vulnerabilities found during penetration testing or suspected compromises. Update the incident response plan if advances in information security technologies demand it. Add a point to the incident response plan that requires the plan to be updated as a reaction to a suspected compromise or penetration testing. Add this same requirement to the information security policy.
