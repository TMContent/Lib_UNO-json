Applies To

  • Web Applications

What to Check For

Verify that debugging data is not included in output.

Why

Debug information may have sensitive information, such as call stacks, fragments of the source code, SQL statements and more. This information may aid an attacker by giving them additional information about the application, especially in buggy areas. Since debugging data is often displayed in error messages, the fact that error messages are being displayed may indicate the presence of a vulnerability, and the debugging data may provide information necessary to exploit the vulnerability.

How to Check

To verify that debugging data is not included in output:

  1. Verify that PHP error messages are disabled. If you are using PHP, examine the php.ini file to make sure the display_errors directive is used to disable PHP error messages, like so: {
  2. Verify that debugging is turned off. If the application has a debugging mode switch, make sure that it is disabled.
  3. Verify the debugging data is not included in error messages. Find all code that generates error messages and review it to make sure that debugging information is not included in the messages. This is the most important step, because attackers often use error messages to look for vulnerabilities.
  4. Verify that any code used to display debugging output is disabled. It is a common practice to include temporary code that generates debugging messages during program execution. If your developers use this practice, review their code for any such code. Usually, a developer will use the same type of code to display debugging output, so a simple string search should be able to find all of it.

How to Fix

To prevent the application from displaying debugging data in output:

  1. Disable PHP error messages. If you are using PHP, use the display_errors directive in php.ini to disable PHP error messages, like so: display_errors = off
  2. Turn off debugging. If the application has a debugging mode switch, disable it.
  3. Do not display debugging information in error messages. Find all code that generates error messages and review it to make sure that debugging information is not included in the messages. This is the most important step, because attackers often use error messages to look for vulnerabilities.
  4. Disable any custom logging functions used for debugging. It is a common practice to include temporary code that generates debugging messages during program execution. All of this code has to be found and disabled, either by commenting it out or removing it.
