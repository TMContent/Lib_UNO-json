What to Check For

Ensure log files are protected.

Why

Log files often contain internal application details that could enable an attacker to more effectively target your application.

How to Check

Use these steps to ensure log files are protected:

  1. Verify that the log location is not disclosed. Scan your application's code base to verify that the location of the log files is known only to the internal logging framework.

  2. Check the file permissions of logs. Verify that the logs are set with the appropriate file system permissions. All entities, excluding your application and its administrators/developers, must be prohibited access to the logs. If your application backs up its logs, verify that log backups are stored into an environment that is as secure as the storage environment for the current logs.

  3. Check for common logging mistakes. Scan your application's code base for common logging mistakes such as:

    • Logging data structures that contain user-supplied data
    • Appending user-supplied data to an exception's message and/or details

    Note: PCI DSS certification requires that all access to the logs is logged, that logs must be promptly backed up to a centralized log server or media that is difficult to alter, and logs should be reviewed regularly to watch for suspicious activity that could indicate access by an attacker.

How to Fix

Use the following steps to protect the application's log files:

  1. Avoid disclosing location of logs. The location of the log files should be known only by your application and its administrators. Avoid disclosing the location of the logs to users.

  2. Set appropriate file permissions. Set the file system permissions so that only your application and its administrators/developers have access to its logs.

  3. Keep the logging framework away from users. Do not allow your application's users to interact directly with the logging framework. The logging framework should log information regarding the performance of the application but should avoid storing user-supplied data inside the logs. Some user-supplied information can aid an attacker into poisoning your application's logs.

  4. Consider using log rotation. Log rotation prevents your application from working with large log files. It allows you to manage and audit your application's logged events without interfering in your application's workings. Use the following steps when utilizing log rotation:

    1. Define a rotation period: Schedule your log rotation on a regular basis. For example, rotate your logs every Monday.

    2. Create a new log file: Create a new log file when your application needs to rotate its log files. Major operating systems allow a file to only be appended to (not read or modified).

    3. Back up the existing log files: Back up the existing log files when your application needs to rotate its log files. Consult your internal auditing policy to determine the period for which logs should be backed up. If your application backs up its logs, store them into an environment that is as secure as the storage environment for the current logs.
      Note: PCI DSS requirements state that logs should be kept for at least one year.

  5. Consider using an external logging application. Using an external logging application ensures that your application's logs will be kept up-to-date even if your application abruptly terminates. Use a custom error handler that calls syslog to deliver error messages, configuring syslog securely. On windows, this goes to the event log.

  6. Consider encoding output that is sent to log files. This can be useful in preventing log forging. Stripping out certain characters &#91;\r\n&#93; that are known to create new lines, which can allow an attacker to forge log entries.
