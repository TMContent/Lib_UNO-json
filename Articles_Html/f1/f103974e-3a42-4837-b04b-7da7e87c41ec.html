Applies To

  • HTML5

What to Check For

Verify that the event.origin property is used to ensure that messages are coming from a trusted domain.

Why

Usually, data needs to be validated on the server side. However, HTML5 applications shift some of the application logic to the client-side, which places some of the data-validation responsibility on the client-side. Check the event.origin property of incoming events to prevent attackers from sending malicious data to your application. If the origin property of incoming messages is not validated, an attacker might be able to cause damage such as making the application display incorrect information, for example, an incorrect balance statement or invoice data.

How to Check

To verify that the event.origin property is used to ensure that messages are coming from a trusted domain:

  1. Identify code that receives messages from the server. Search application code for all references to the addEventListener method. Create a spreadsheet to list all code that receives messages from the server.
  2. Define trusted domains. In the spreadsheet, for each piece of code that uses addEventListener, add information about domains that are authorized to send messages to the application. If these domains might change after the application is deployed, it may be prudent to include a configurable option in the application to choose trusted domains, or to add logic that will calculate trusted domains based on the environment. Add all of this information to the spreadsheet.
  3. Check code that validates the origin of messages. Review each piece of code that receives messages from the server to make sure that before the messages are processed, the event.origin property is used to validate the message origin. If it is necessary to calculate the trusted domain(s) based on data available at run-time, or to make choosing the trusted domain(s) a configurable option in the application, review the code that is responsible for that functionality.

How to Fix

To use the event.origin property to ensure that messages are coming from a trusted domain:

  1. Identify code that receives messages from the server. Search application code for all references to the addEventListener method. Create a spreadsheet to list all code that receives messages from the server.
  2. Define trusted domains. In the spreadsheet, for each piece of code that uses addEventListener, add information about domains that are authorized to send messages to the application. If these domains might change after the application is deployed, it may be prudent to include a configurable option in the application to choose trusted domains, or to add logic that will calculate trusted domains based on the environment. Add all of this information to the spreadsheet.
  3. Add code that validates the origin of messages. For each piece of code that receives messages from the server, add code that validates the event.origin property to match the trusted domain(s) listed in the spreadsheet. If it is necessary to calculate the trusted domain(s) based on data available at run-time, or to make choosing the trusted domain(s) a configurable option in the application, add the code for doing this as well.

Web Messaging Guidelines

  • Do Not Evaluate Messages as Code
  • Use element.textContent to Assign Values to DOM Elements
  • Explicitly State the Expected Origin with postMessage
  • Validate Data Received by Event Handlers
  • Validate the Origin Attribute Against the FQDN
  • Check The Origin Attribute of Messages
