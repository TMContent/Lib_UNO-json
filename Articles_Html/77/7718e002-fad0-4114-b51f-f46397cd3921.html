Applies to

Scala and Play Framework 2.1.0+

What to Do

Do not place sensitive data in unprotected parameters. Sensitive data should either be stored in sessions or encrypted before being sent to the server in parameters.

Why

Sensitive data stored in unprotected parameters is sent on the network in plain-text and can be easily stolen and read&nbsp;by an attacker with a network sniffer.

When

If your application accepts sensitive data as input or uses it to track session information, do not place that data in unprotected parameters.

How

Follow these steps when handling sensitive data via parameters:

  1. Identify the parameters.&nbsp;Parameters can be used both as entry points and storage places. Locating your application's parameters helps identify potential sources of input. Because parameters are easily accessible by the client, they should be considered untrusted. Different types of parameters include:

    • URL-based parameters
    • Form-based parameters
    • Hidden fields
    • Cookies
  2. Identify the use.&nbsp;Parameters can be used for several purposes. If you are using parameters to send sensitive data, use the steps below to find a work-around:&nbsp;

  • Submitting input: Submitting sensitive data using parameters should always be done through an encrypted channel and always using a POST request. Use TLS to ensure encryption. Avoid using hidden fields when gathering information from the user. Allowing the user to enter all information about their account ensures that they are aware of all data that is being sent. Any information regarding the user's client environment can be gathered via the HTTP headers.
  • User tracking: User tracking should always be done server-side using sessions. Avoid storing sensitive information in hidden fields and cookies. If it is necessary to store sensitive data using parameters, encrypt the data using a key that is known only by your application. If encrypted sensitive data is stored inside a cookie, avoid setting the cookie as persistent.

Problem Example

In the following page, the credentials are sent using a GET request. When the page is submitted, the credentials will be available in the URL. Therefore, it is possible for a malicious user to obtain the credentials via the server's access logs or by sniffing the HTTP referrer header.

Code for Loginform.scala.htm:

@main(title) {
 @helper.form(action = routes.Application.authenticateUser) {
        
        <fieldset><legend>@title</legend>
                       
            @inputText(
                loginForm("email"), '_label -> "Email ",
                '_help -> "Enter a valid email address."
            )
            
            @inputPassword(
                loginForm("password"), 
                '_label -> "Password",
                '_help -> "A password must be at least 6 characters. "
            )          
        </fieldset><div class="actions"><input type="submit" class="btn primary" value="Sign in " /><input type="hidden" name="hide" value="1234" />
               Or
            <small><a href="@routes.Application.siginUpForm">Sign Up </a></small></div>
        
       }     
}
Code for /conf/routes: 
# Home page
GET     /                           controllers.Application.index
GET    /printMessage		    controllers.Application.printMessage
GET   /login			    controllers.Application.authenticateUser
GET /javascriptRoutes               controllers.Application.javascriptRoutes
&nbsp;

Solution Example

In the following page, the credentials are sent using a POST request. When the page is submitted, the credentials will not be shown in the URL. The only change to be made in our case is that the routes configuration should change the method for "login" from GET to POST.

Code for /conf/routes: 
# Home page
GET     /                           controllers.Application.index
GET    /printMessage		    controllers.Application.printMessage
POST  /login			    controllers.Application.authenticateUser
GET /javascriptRoutes               controllers.Application.javascriptRoutes
