What to Check For

Ensure that debug information is not disclosed to the client.

Why

Debug information may have sensitive information, such as call stacks, fragments of the source code and more. This information may aid an attacker by giving them inside-information.

How to Check

Debug information is often displayed in error messages during development in order to speed up the debugging process. After deployment, such output can greatly aid an attacker by revealing details about your application that normally would not be accessible. In order to prevent such information disclosure, verify the following:

  1. Ensure that proper exception handling is used. Proper exception handling will prevent debug and source data from being disclosed to the user and allow your application to exit gracefully. There are two types of exception handling mechanisms that can be used:
    • Dynamic handling: Verify that the root execution code is executed inside a traditional try/catch/finally block.
    • JSP error pages: Verify that error pages are defined for each exception type (making certain there is one covering java.lang.Throwable) as well as any applicable HTTP error codes in your application's web deployment descriptor (e.g. web.xml) using a <location> tag that points to a JSP error page with the isErrorPage attribute set to true.

  2. If necessary, use a logging mechanism to collect debug information. If it is necessary to collect debug information for later analysis, use a logging mechanism to reduce the risk of an attacker gaining access to the information.

    • Consult your application's designs to determine the appropriate framework for logging debug messages.
    • Identify all catch and finally clauses, and ensure that debug messages are not sent to the client. Ensure error messages are generic and all necessary debug information is collected via your application's logging framework.

How to Fix

The purpose of debug information is to identify and troubleshoot potential problems within your application. Use the following steps to ensure debug information is not transmitted to the client:

  1. Collect debug information via a logging mechanism. Use a logging mechanism instead of transmitting debug information through the HTTP response to the client.

  2. Use proper exception handling. There are two approaches to placing an exception handler that allows your application to exit gracefully and hide debug and source data from the user:

    • Dynamic Handling: This allows the handling to be conducted inside the Java code. It consists of placing the entire root execution code inside a try/catch/finally block. Using dynamic handling allows your application to log the error and provide an error message to the user without having to disclose any detailed information about the raised exception. Example: 

      public void doGet(HttpServletRequest request,
            HttpServletResponse response) throws ServletException, IOException
      &#123;
            PrintWriter out = response.getWriter();
            workDone = false;

            try
            &#123;
                  ...
            &#125;
            catch (Exception e)
            &#123;
                  // Add appropriate logging and exception handling mechanisms
                   out.println("An error has occurred.");
            &#125;
            out.flush();
            out.close();
      &#125;

    • JSP Error Pages: JSP error pages provide an effective way of telling the user that something went wrong. The location of the appropriate error page is specified in your application's web deployment descriptor (e.g. web.xml) using the <location> tag.

      For example:

      <&#33;-- Catch a system error using an HTML page -->
      <error-page>
        <exception-type>java.lang.NullPointerException</exception-type>
        <location>/NPEerror.html</location>
      </error-page>

      Ensure that there are error pages written for all specific exceptions that you need to handle, including one that covers java.lang.Throwable. This will catch all previously unhandled exceptions and redirect the user to the specified error page. In addition to handling exceptions, you should also handle HTTP error codes. An example is below.

      <ERROR-PAGE><ERROR-CODE><error-page>
      &nbsp;&nbsp; <error-code>404</error-code>
      &nbsp;&nbsp; <location>/error/error404.jsp</location>
      </error-page>&#9;</ERROR-PAGE>

      The JSP error pages should begin with the <&#37;&#64;page &#37;> tag.

      For example:

      <&#37;&#64;page isErrorPage="true" &#37;>

      Creating error pages for specific exception types not only allows you to ensure sensitive data is not sent to the client, but it also improves the user experience. Design your error pages so they are useful for a typical user (not a developer), and you are unlikely to include any information that an attacker will find useful.
