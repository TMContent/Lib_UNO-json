What to Do

When exceptions occur, return concise error messages to the client and log specific details on the server. Do not reveal internal system or application details, such as stack traces, SQL statement fragments, and table or database names to the client. Ensure that this type of information is not allowed to propagate to the end user or beyond your current trust boundary.

Why

A malicious user could use system-level diagnostic information to learn about your application and probe for weaknesses to exploit in future attacks.

When

This guideline is applicable to any application pages capable of generating errors.

How

Prevent detailed error messages from displaying by setting the mode attribute of the <CustomErrors>&nbsp;<CUSTOMERRORS>element to On, so that all callers receive filtered exception information. Do not use mode="Off" because this allows detailed error pages intended for application developers that contain system-level information to be returned to the client.&nbsp;

You should also use the <customErrors> section of the Web.config file as shown in the following code example to specify a default error page to display, along with other required error pages for specific HTTP response codes that indicate errors.

<customErrors mode="On" defaultRedirect="ErrDefault.aspx">
&nbsp;&nbsp;&nbsp; <error statusCode="401" redirect="ErrUnauthorized.aspx" />
&nbsp;&nbsp;&nbsp; <error statusCode="404" redirect="ErrPageNotFound.aspx" />
&nbsp;&nbsp;&nbsp; <error statusCode="500" redirect="ErrServer.htm" />
</customErrors>

The defaultRedirect attribute allows you to use a custom error page for your application, which. for example, might include support contact details. Use these application-wide error pages to provide user-friendly responses for errors that are not caught in a structured event handling.&nbsp;

Problem Example

The developers of an ASP.NET application change the mode attribute of the <CustomErrors>&nbsp;<CUSTOMERRORS>element to Off in order to display detailed error messages on both the local machine and remote clients for debugging purposes.&nbsp; When the application goes into production this setting remains and detailed error messages are displayed to all users.

Solution Example

During development an ASP.NET application sets the mode attribute of the <CustomErrors>&nbsp;<CUSTOMERRORS>element to RemoteOnly to aid debugging by displaying error messages on the local machine. Once the application is deployed the mode attribute is set to On to ensure that detailed error messages are not sent to any client.&nbsp; The defaultRedirect element is used to specify a generic error page while <error> elements are used to specify more specific error pages.

<customErrors mode="On" defaultRedirect="ErrDefault.aspx">
&nbsp;&nbsp;&nbsp; <error statusCode="401" redirect="ErrUnauthorized.aspx" />
&nbsp;&nbsp;&nbsp; <error statusCode="404" redirect="ErrPageNotFound.aspx" />
&nbsp;&nbsp;&nbsp; <error statusCode="500" redirect="ErrServer.htm" />
</customErrors>
