##Applies To##

  • Amazon Web Services (AWS)

##What to Do##

Review samples of security code, and develop custom, reusable code for use in multiple applications. Submit your code for testing against known threat agents. Use the change control process specified in your security policy for storing code in a security code library.

##Why##

Reusing security code creates a more-secure application for these reasons:

  • reduces code complexity and associated maintenance costs
  • ensures security requirements are addressed with proven controls
  • prevents re-introduction of bugs and vulnerabilities
  • allows testing of the security code in isolation
  • allows for performance tuning while maintaining security posture
  • permits offloading of security processing to network appliances.

##When##

When working with various programming and scripting languages, create new versions of reusable security code as required for each programming language, such as Java, PHP, Python, Ruby and .NET. This includes routines for authentication, cryptographic, and logging code, for example.

##How##

The development of security code will generally parallel your software development lifecycle, and will follow these general steps.

  1. Identify requirements and perform analysis. Identify requirements which this security code must address. Build use cases to identify user, required privileges, protocol, and messages.

  2. Design software architecture. Develop specific code parameters to work within the application architecture boundary. Include inputs and outputs from the requirements specification. Use data flow diagrams and other illustrations to transform use cases into pseudo code.

  3. Write software code. Actual coding is developed at this time. This is the longest phase of the cycle.

  4. Test security code. Test the security code against the requirements by using standard test methods.

  5. Deploy security code. Add the security code to current projects.

  6. Maintain software library. When the security code needs updating due to deficiencies noticed in actual use, use change control procedures to revise or add requirements. Submit the code for security testing, and add to the code security library when your change control sign-offs are completed.
