What to Do

Whenever possible you should use an exception management strategy for error handling. If this is not possible due to compiler or language limitations then use error return codes when necessary.

Why

If your function has an error that can impact your caller and you do not notify the caller they can do nothing to handle this problem. The result could be an application crash or other denial of service problem. In an extreme condition it could result in erroneous application logic that may be exploited by a determined attacker.

When

If your function can fail in a way that impacts the logic of your caller and can have a detrimental effect on the running of your application, return an error code to notify the caller.

How

Check for error conditions in your code and return an error code to your caller when necessary. It is very common for an integer return value of -1 to be an error code. It is also common to use a bool return value and return false in the event of an error. You may also use SetLastError in Win32 if you need to specify additional error details.

Problem Example

In the example below the function does not check for an error and does not return an error code in the case of failure of the read call.

bool ReadFromSocket(SOCKET socket, char *recvBuffer, int len)
{
  //first read from the socket
  read(socket, recvBuffer, len, 0);


  //now parse the buffer
  return parseBuffer(recvBuffer, len);
}

Solution Example

In the following example, the read API is called and then the return value is checked for errors. The function will return an error code if the read call fails.

bool ReadFromSocket(SOCKET socket, char *recvBuffer, int len)
{
  //first read from the socket
  int ret = read(socket, recvBuffer, len, 0);


  if (ret >= 0)
  {
      //now parse the buffer
      return parseBuffer(recvBuffer, len);
  }
  else
  {
      //can call WSAGetLastError if we need error details
      return false;
  }
}
